## CHS（Cylinder-Head-Sector）
早期磁盘的物理寻址方式：
- Cylinder：柱面号（哪个同心圆环）
- Head：磁头号（哪个盘面）
- Sector：扇区号（磁道上的第几个扇区）
依赖磁盘的物理几何参数，BIOS 用 24-bit 表示（C:10bit, H:8bit, S:6bit），最大寻址 ~8.4GB。
## LBA（Logical Block Addressing）
逻辑扇区号，从 0 开始线性递增，不关心物理位置：
- 磁盘固件内部维护 LBA → (C,H,S) 映射表
- 操作系统/文件系统只管 LBA 编号，参数要求（如 28-bit LBA ≈ 128GB，48-bit LBA ≈ 128PB）
- 1990s 后期 ATA-1 引入，随大容量磁盘淘汰 CHS
## 为什么LBA比CHS寻址大
纯位数问题，不是方案本身的优劣：
- CHS（ATA）：C 10bit × H 8bit × S 6bit = 24bit → 最大约 8.4GB（512B/sector）
- LBA 28-bit：2^28 × 512B ≈ 128GB
- LBA 48-bit：2^48 × 512B ≈ 128PB
CHS 的 24bit 是 BIOS/ATA 标准当年定的死上限。LBA 只要协议升级就能扩展位数，从 28bit 到 48bit 再到未来的 64bit，没有物理几何参数的天花板

## 伺服信息（Servo Information）
伺服信息（Servo Information）是出厂时写入盘片的永久磁标记，用于磁头定位，不存储用户数据。
物理形态
每隔一定角度（如每 60° 或每个扇区间隙），在盘片上写入一段伺服楔（Servo Wedge），形成从圆心到边缘的辐射状条纹

伺服楔内的数据
一段完整的 Servo Wedge 包含多个字段（按时间顺序读到）：

| 字段                      | 内容          |
| ----------------------- | ----------- |
| Preamble                | 固定频率信号      |
| Servo Address Mark(SAM) | 特定位模式       |
| Gray Code               | 格雷码编码的 磁道号  |
| Burst Pattern (A/B/C/D) | 四组相位偏移的交变信号 |

工作流程
1. 粗定位：移动磁头→读取 Gray Code 确认磁道号
2. 精定位：读 A/B/C/D Burst 信号强度对比，算出磁头偏离磁道中心的偏移量 → 通过 VCM（音圈电机）伺服回路微调
3. 保持跟踪：旋转过程中周期性地读伺服楔（每转几十~几百次），不断修正

伺服楔（Servo Wedge） 和 扇区（Sector） 是两个不同的划分维度：
伺服楔 ≈ 径向辐射线（用于磁头定位），典型数量是 每转 60~200 个，也就是每 6°~1.8° 一个。60° 指的是伺服楔。
扇区 是磁道上的弧段，数量按磁道算（ZBR 下外圈 ~1000 个/道，内圈 500 个/道），每个约 0.36°0.72°。两个伺服楔之间可以包含多个扇区：
`[Servo][Sector1][Sector2][Sector3]...[Servo][Sector1][Sector2]`...
伺服楔是径向对齐的（所有磁道在同一角度位置都有伺服楔），而扇区边界在各磁道上不需要对齐。

# 簇 / 块 （cluster / block）

- cluster 簇 windows的叫法
- block 块 Linux的叫法

簇 = 文件系统一次分配的最小空间。一个 1 字节的文件在 4KB 簇的 FS 上也占 4KB 磁盘空间（空间碎片），而 I/O 也是按簇对齐的

# block vs sector

Sector 是硬件/设备的最小寻址单元（通常 512 字节或 4K），由磁盘物理决定，驱动层面使用。
Block 是文件系统/VFS 的逻辑抽象单元（通常是 sector 的 2 的幂次倍，如 1024/2048/4096 字节），由文件系统格式化时决定，块层配合处理。
关键关系：
- 块层内部通过 blksize_bits / sector_t 做转换
- struct bio 中的 bi_sector 以 sector 为单位
- struct buffer_head / struct page 缓存以 block 为单位
- 一个 block 必然对应整数个 sector（通常 1 block = 2^n sectors）

举例：
假设你有一个 4K 物理扇区的 NVMe 盘，格式化为 ext4 块大小 4K：
磁盘 LBA 0 上的 "第 8 个文件系统块":
  文件系统层: block number = 8, 每个 block = 4096 bytes
              文件偏移 = 8 × 4096 = 32768

  传递到块层: 需要换算成 sector
              sector = block × (blocksize / SECTOR_SIZE)
                      = 8 × (4096 / 512)
                      = 8 × 8
                      = 64

  bio 中:     bi_sector = 64   ← 固定以 512B 为单位
              实际读盘: 告诉控制器从 LBA 64 开始读 8 个 sector
              (盘片固件收到后合并成 1 个 4K 物理操作)
内核源码里的转换（fs/buffer.c 随处可见）：

```
// sector = block << (blocksize_bits - SECTOR_SHIFT)
sector_t blocknr = bh->b_blocknr;   // 文件系统块号 = 8
sector_t sector = blocknr << (inode->i_blkbits - SECTOR_SHIFT);
// i_blkbits=12 (4096), SECTOR_SHIFT=9 → 左移 3 位 → sector=64
```

所以同一个数据，在文件系统里叫 block 8，进了块层变成 sector 64，本质是因为单位不同——block 可变，sector 永远 512B。

### 块层（Block Layer） = block/ 目录下的内核子系统，是 VFS/文件系统和块设备驱动之间的通用 I/O 中间层
核心职责就是处理 struct bio → struct request → 驱动的整条路径：
文件系统 / 直接 I/O
             ↓ submit_bio(bio)
┌────────────────────────────┐
│      块层 (block layer)                                 │
│  bio合并/拆分  I/O调度                               │
│  blk-mq多队列分派 限流                             │
│  分区管理 错误处理                                     │
└──────────┬─────────────────┘
               ↓ blk_mq_submit_bio() → 驱动队列
   NVMe / AHCI / virtio-blk 等驱动
入口是 `block/blk-core.c:submit_bio()`，核心数据流：bio → 调度器合并为 request → blk-mq 分派到驱动


# 扇区 vs 块/簇 vs Page

三个不同层的数据单元，通过 2 的幂次搭桥：

| 层级     | 单位     |
| ------ | ------ |
| 磁盘/块层  | Sector |
| 文件系统   | Block  |
| MM/页缓存 | Page   |

映射关系（4KB 场景）：
     1 Page =    1 Block         =     8 Sectors
        ↓               ↓                          ↓
   PAGE_SIZE    BLOCK_SIZE    SECTOR_SIZE
   (MM管理)   (FS分配单位)   (块层寻址)
