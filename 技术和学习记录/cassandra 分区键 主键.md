# 主键、分区键、聚类键

**1. 主键 (Primary Key)**
CQL 中的 `PRIMARY KEY` 是一个总称，由**分区键 + 聚类键(可选)** 组成，用于唯一标识一行。

**2. 分区键 (Partition Key)**
主键的**第一部分**，决定数据存到哪个节点/分区。同一分区键的数据物理上存储在一起。分区键只决定"在哪",不决定行内排序。

**3. 聚类键 (Clustering Key)**
主键中分区键**之后**的列，决定同一分区内行的排序顺序，可在分区内唯一标识行。

---

**关系与写法对比：**

```sql
-- 单列主键：user_id 既是分区键又是主键
PRIMARY KEY (user_id)

-- 复合主键（联合主键）：user_id 是分区键，created_at 是聚类键
PRIMARY KEY (user_id, created_at)

-- 复合分区键：(user_id, bucket) 共同作为分区键，created_at 是聚类键
PRIMARY KEY ((user_id, bucket), created_at)
```

**要点：**
- `PRIMARY KEY (a, b, c)` → 括号里**第一个** `a` 是分区键，其余 `b, c` 是聚类键（不是联合主键意义上的"三个一起当分区键"）。
- 想让多列共同作为分区键，必须再加一层括号：`((a, b), c)`。
- 分区键**不可变**（改分区键等于换分区），聚类键也属于主键的一部分同样不可更新。
- 查询时**必须**先按分区键等值定位（`=` 或 `IN`），再按聚类键范围/排序过滤，否则报 `ALLOW FILTERING`。


# 什么情况下需要复合分区键

需要**两个（或多个）字段作为复合分区键**的场景，核心目的通常是：**避免单分区过大、保证数据均匀分布、匹配查询模式**。

**1. 防止分区过大（hot partition / wide row）**
单一分区键下数据量太大，单个分区可能超过 Cassandra 建议的 ~100MB / 10万行上限。

例：日志表若只用 `user_id` 做分区键，活跃用户几年日志全挤一个分区 → 加时间桶：
```sql
PRIMARY KEY ((user_id, day), ts)
```

**2. 保证负载/数据均匀分布**
分区键基数太低或分布倾斜（如按 `country` 分区，中国/美国分区巨大），加一个高基数列打散：
```sql
PRIMARY KEY ((country, bucket_id), ...)
```

**3. 匹配查询模式**
业务总是"先按 A 再按 B"定位，两者一起才能缩小到合理范围，且需要同分区共置数据：
```sql
-- 按店铺+日期查订单明细
PRIMARY KEY ((shop_id, order_date), order_id)
```

**4. 实现限流/分页桶**
把大表人为切分成固定大小的"桶"，复合键做桶标识：
```sql
-- feed 流分页，每桶 N 条
PRIMARY KEY ((user_id, bucket_no), created_at)
```

---

**取舍：**
- 复合分区键 = 多列**共同**算 hash 决定分区位置，查询时这几列**必须全部用 `=` 等值条件**，少一个都定位不到分区。
- 所以只有"这几个字段总是一起作为查询入口"时才适合用复合分区键，否则应把高频等值列放分区键、其余放聚类键。
