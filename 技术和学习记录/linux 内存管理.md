# 页框（Page Frame）
  是物理内存的划分单位。物理内存被等分成固定大小的块，每块就是一个页框（也叫物理页）。常见大小为
  4KB。页框是硬件层面的概念，是内存管理单元（MMU）寻址物理内存的基本单位。

#  页表（Page Table） 
  是操作系统维护的数据结构，用于建立虚拟页号 →
  物理页框号的映射关系。进程使用的是虚拟地址，MMU 通过查页表将其翻译成物理地址，才能访问实际的内存页框。

  关键点
  - 页是虚拟内存的概念，页框是物理内存的概念，大小相同（通常 4KB）
  - 页表存放在内存中，由 OS 维护、MMU 硬件查询
  - 为了加速查表，CPU 有 TLB（Translation Lookaside Buffer），缓存最近使用的页表映射
  - 页表项（PTE）除了页框号，还包含权限位（读/写/执行）、存在位、脏位等控制信息

通常一个页表是一页的大小，即4KB =  $2^{12}$ = 4096

# 页表项（Page Table Entry, PTE）

  简单总结：页表项 = 物理页框号 + 一组控制标志位，是虚拟内存系统的核心数据单元。
一共64bit，即8Byte = $2^3$ 

正常来说1页能放下多少页表项呢？ 4096 / 8 = $2^9$
为啥非要用2的多少次方来展示，其实是9位的地址可以表达这么多的页表项的意思
x86_64系统经典的4级页表，实际上的虚拟内存地址是48位（9+9+9+9+12）

# 二进制数学知识
  4KB = 4096 = 2^12

  一个地址如果是 4096 的整数倍，那么二进制低 12 位一定是0。(为0的这3位就可以表示4096)
0x00 , 这里的一个0这一位是一个16进制数（0-F），1个16进制数可以用4位来表示（$2^4$）,所以这里<mark style="background:#fff88f">1个进制位 = 4bit</mark>

![[Pasted image 20260519205702.png]]
12bit可以表示 0 - 4095 共4096个数
3bit可以表示0 - 7共8个数

  例如：
  0x12345000
  低 12 位就是：
  0x000

  所以页表项里保存下一级页表地址时，不需要保存低 12 位，因为它们反正都是 0。

  于是页表项可以这样用：

  高位：保存下一级页表/物理页框的基地址
  低位：保存 present、writable、user、dirty 等标志位

  
  页表项里存的是 4KB 对齐的物理页地址，所以地址低 12 位天
  然为 0，硬件就利用这些低位存页表标志位。

# 多级页表的原因


```
• 关键点是：多级页表不是把单级页表平均拆成几份就省内存，而是“没用到的分支根本不创建”。

  假设一个极简地址系统，不用真实 x86，方便看清楚原理。

  虚拟地址一共 16 bit，页大小 256B = 2^8，所以：
  虚拟地址 = 8 位虚拟页号 + 8 位页内偏移
  也就是说，一个进程最多有：

  2^8 = 256 个虚拟页

  每个页表项假设 4B。

  单级页表

  单级页表必须长这样：

  页表[0]
  页表[1]
  页表[2]
  ...
  页表[255]

  一共 256 项。哪怕进程只用了 2 个虚拟页，也要有 256 个表项，因为 CPU 要能直接用“虚拟页号”当数组下标查：

  页表[虚拟页号]

  所以页表大小是：

  256 * 4B = 1024B

  假设这个进程只用了两个虚拟页：

  虚拟页 3
  虚拟页 200

  单级页表仍然要保留完整 256 项：

  [0 空]
  [1 空]
  [2 空]
  [3 有效]
  [4 空]
  ...
  [200 有效]
  ...
  [255 空]

  大量表项只是“空”。

  两级页表

  现在把 8 位虚拟页号拆成两段：

  高 4 位 = 一级索引
  低 4 位 = 二级索引

  于是：

  一级页表有 16 项
  每个二级页表有 16 项

  查虚拟页号时：

  一级索引 -> 找到某个二级页表
  二级索引 -> 找到最终页表项

  如果虚拟页号是 3：

  3 = 0x03
  一级索引 = 0
  二级索引 = 3

  需要：

  一级页表[0] -> 指向二级页表 A
  二级页表 A[3] -> 有效

  如果虚拟页号是 200：

  200 = 0xc8
  一级索引 = 12
  二级索引 = 8

  需要：

  一级页表[12] -> 指向二级页表 B
  二级页表 B[8] -> 有效

  注意：一级页表的其他项，比如 [1] [2] [3] ... [11] [13] ... [15]，都没有被用到，所以它们不需要指向任何二级页表。

  于是实际分配的是：

  一级页表：16 项
  二级页表 A：16 项
  二级页表 B：16 项

  总共：

  (16 + 16 + 16) * 4B = 192B

  对比单级页表：

  单级：1024B
  两级：192B

  为什么省了？因为没有创建这些二级页表：

  一级[1] 对应的 16 项
  一级[2] 对应的 16 项
  一级[3] 对应的 16 项
  ...

  它们对应的虚拟地址范围根本没用到，所以不存在。

  这就是多级页表省内存的本质：

  单级页表：
  必须为每一个虚拟页准备表项

  多级页表：
  只为用到的虚拟地址范围创建下级页表

  换成真实 x86-64：

  48 位虚拟地址
  4KB 页
  每项 8B

  单级页表需要覆盖所有虚拟页：

  2^(48-12) = 2^36 个页表项
  2^36 * 8B = 512GB

  但四级页表里，每一级页表页只有 4KB，一页 512 项。假设进程只映射了一个 4KB 页面，理论上只需要创建一条路径：

  PGD 页
  PUD 页
  PMD 页
  PTE 页

  总共约：

  4 * 4KB = 16KB

  而不是：

  512GB

  所以你抓住这一句就行：

  分级本身不神奇，真正省内存的是：上级页表项为空时，整片下级页表都不用分配。
```

# 1页是4KB

● 4KB 是一个历史选择，是多种因素权衡的结果：

  太小（比如 1KB）的坏处：
  - 页表项变多，页表占用内存大
  - 磁盘 I/O 次数增多，每次只读写很少的数据
  - TLB 能缓存的地址范围变小，命中率下降

  太大（比如 1MB）的坏处：
  - 内部碎片严重。进程可能只需要 100 字节，但也得分配一整页
  - 换页时单次 I/O 数据量大，延迟高

  4KB 是当年的甜点值：
  - 对磁盘扇区（512B）是整数倍，对齐方便
  - 足够装下大部分常用数据结构（一个页表本身刚好 4KB）
  - 内部碎片浪费可接受（平均浪费 2KB）

  另外这也不是唯一的页大小，x86_64 支持 2MB 大页（Huge Pages），Linux 也支持 1GB
  大页，用于数据库等需要大内存、减少 TLB miss 的场景


# 不同进程里相同的虚拟地址，完全可能映射到不同的物理地址

  比如：

  进程 A:
  虚拟地址 0x400000 -> 物理地址 0x12345000

  进程 B:
  虚拟地址 0x400000 -> 物理地址 0x8abc9000

  原因是每个进程通常有自己的页表。CPU 当前运行哪个进程，就
  使用哪个进程的页表，所以同一个虚拟地址会按不同页表翻译出
  不同物理地址。

  但也有例外：不同进程的相同虚拟地址也可能映射到同一个物理
  页，比如：

  共享库的只读代码页
  mmap 共享内存
  fork 后的 copy-on-write 页面
  内核映射区

观点： 用户态虚拟内存空间 是针对进程为单位来说的
