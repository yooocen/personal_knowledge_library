# 为什么需要块组

假设你有一个 **1TB 的 ext4 分区，块大小 4KB**。

### 如果没有块组，整个分区只有一个位图

- 总块数：`1TB / 4KB = 256,000,000` 块
- **块位图大小**：`256,000,000 / 8 = 32MB`
- **inode 位图**：如果 `inode = 256 bytes`，`s_inodes_per_group × s_inode_size` 推算，inode 表也需要巨大连续空间
- **inode 表**：如果总共 256M 个 inode，每个 256 字节，inode 表 = **64GB**

### 具体问题

**1. 分配一个块就要读 32MB 位图**

你要写一个文件，内核需要找个空闲块。没有块组，就得把 32MB 的位图从磁盘读到内存，逐 bit 扫描才能找到空闲块。一次 `open()` / `write()` 触发几十 MB 的 IO —— 性能灾难。

有块组后，块组大小 = `blocks_per_group × block_size`（默认 32768 × 4KB = 128MB），块位图 = `32768 / 8 = 4KB`，刚好一个块。分配时只读一个块（4KB），快 8000 倍。

**2. inode 表全是空洞**

某组里创建了 10 个文件，但 inode 表是全局连续的 64GB 数组。VFS 要加载 inode #500 时，得从那 64GB 线性区域里的某个位置读 —— 即便只需要一个 256 字节的 inode。没有 locality，全是随机跳。

有块组后，inode #500 落在固定块组，inode 表就几个 block 大，加载快。

**3. 无法并行**

全局就一把锁保护位图。你给 100 个文件并发写入，它们全在抢同一张 32MB 位图，**CPU 全耗在锁竞争上，磁盘一个挨一个串行查位图**。

有块组后，不同块组有独立位图和锁，100 个文件分配到 100 个不同组，互不干扰。

**4. 超级块损坏 = 全盘死**

没有块组=没有冗余。超级块所在的 block 坏了，整个分区认不出来。有块组后，超级块在多个块组有副本，主超级块坏了可以从副本恢复。


# 位图

位图（bitmap）是一种**空间管理的数据结构**，本质就是一个比特数组，每个比特（0 或 1）表示一个资源（块或 inode）是否被占用。

### 在 ext4 中

每个块组有两个位图：

**1. 块位图（Block Bitmap）** — 1 block = 4KB = 32768 bit
- 每个 bit 对应块组里的一个**数据块**
- **1** = 已占用，**0** = 空闲
- 要分配一个块时，扫描位图找第一个 0，设为 1，然后写入数据

```
块位图 (4KB = 32768 bits):
bit 0: 1  (块0 已用)
bit 1: 1  (块1 已用)
bit 2: 0  (块2 空闲) ← 分配从这里拿
bit 3: 1  (块3 已用)
...
bit 32767: 0 或 1
```

**2. inode 位图（Inode Bitmap）** — 也是 1 block
- 每个 bit 对应 inode 表里的一个 **inode 条目**
- 同样的 0 = 空闲，1 = 已用

### 为什么用位图

- **空间效率**：1 个 bit 管 1 个块。4KB 的位图可以管理 `4KB × 8 = 32768` 个块，对应 `32768 × 4KB = 128MB` 空间。管理开销只有 **0.003%**。
- **操作简单**：`test_bit()`、`set_bit()`、`find_first_zero_bit()` 都是 CPU 单指令或内置函数，极快。

相比链表或 B 树来管理空闲空间，位图在**小粒度管理下的内存开销最低**。

块组中有上述的位图来表示，一个bitmap的占用空间是 1 block， 1 block = 4KB
所以可以表示的block的数量是 1 block size * 8
1B = 8bit
位图的每一位代表一个block

# 一个块组包含多少inode
[[inode]]
1. inode 位图也是 1 block，最多表示 block_size × 8 = 32768 个 inode
2. inode 表占连续多 block（8192 inode × 256 字节/inode = 2 MB = 512 blocks），不能超出 group 的数据块范围

每个 block group 大小 = 32768 blocks × 4096 字节 = 128 MB
默认 bytes_per_inode = 16384 字节（每 16 KB 空间配 1 个 inode）

inodes_per_group = 128 MB / 16 KB = 8192
如果 inodes_per_group = 32768（用满位图），意味着：
- 每组 inode 表大小 = 32768 × 256 字节 = 8 MB
- 平均每 4 KB 空间就有一个 inode
绝大多数文件远大于 4 KB，会浪费大量磁盘空间在 inode 表上，得不偿失。所以 mkfs 通过 bytes_per_inode 参数（-i）控制 inode 密度，默认 16384 是经验平衡值。
